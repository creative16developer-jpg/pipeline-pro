We need to refactor the pipeline system to support multiple pipeline runs per store, queueing, review flow, and a proper monitoring UI.

---

## 1. CORE ARCHITECTURE

- Each Fetch Job belongs to a store (store_id required)
- Each Fetch Job = ONE pipeline run
- Each pipeline run is stored as PipelineJob

Example:
Store 1:
- Fetch Job 1 → Pipeline PL-001
- Fetch Job 2 → Pipeline PL-002
- Fetch Job 3 → Pipeline PL-003

So:
- Multiple pipelines per store ✅
- Full history maintained ✅

---

## 2. PIPELINE RUNNER PAGE (/pipeline)

This page is ONLY for starting a pipeline.

Flow:
1. Select Store
2. Select Fetch Job (filtered by store)
3. Click "Run Pipeline"

Validation:
- Cannot run without store + fetch job

On Run:
- Create new PipelineJob
- Status = running OR queued (see queue logic below)

## 7. REVIEW STEP (CRITICAL)

After content generation:

- Pipeline status = REVIEW
- Pipeline pauses automatically

Show stats:
- Total
- OK
- Fallback
- Failed

Example:
52 total | 47 OK | 3 fallback | 2 failed

User actions:
- "Resume" → continues to Upload
- "Cancel" → stops pipeline

---

## 8. QUEUE SYSTEM (VERY IMPORTANT)

Rule:
- Only ONE pipeline can RUN per store at a time

If user starts another:
- Status = QUEUED

Behavior:
- Auto-start when running pipeline finishes
- NO manual action required

UI:
- Show text under status:
  "auto-starts when running finishes" (italic)

- Show banner:
  "2 pipelines queued for Store 2 — waiting for PL-012 to finish"

User can:
- Cancel queued pipeline
